# Soft Party UI 优化计划书

## 目标

本计划针对当前 `public/settlement/soft-party/app-preview.html` 的视觉方向进行优化。

当前预览页已经具备不错的亲和力、完成度和统一感，但整体更像“软萌休闲派对 App”，棋盘策略游戏的气质还不够强。优化目标是：**保留温柔、轻松、可爱的基础，同时增强棋盘中心感、策略感、移动端可用性和组件落地稳定性。**

## 当前设计评估

### 做得好的地方

- 蓝白软玻璃视觉干净、友好，和现有产品基础比较贴合。
- 模式卡片、按钮、设置面板的层级清楚，用户知道下一步该做什么。
- 整体风格轻松，没有传统棋类产品的压迫感。
- 模式选择、对局页、联机大厅、段位页、结算页已经形成一套统一方向。
- 浅色卡片风格比重度暗黑、霓虹电竞风更适合当前项目。

### 主要问题

- 对局页里棋盘还不够像视觉主角。
- 风格偏“可爱”，但五子棋还需要一定“策略感”和“对弈感”。
- 移动端首屏密度需要优化，顶部标题和标签占用过多空间。
- 部分结构性图标偏 emoji 风格，不同平台显示可能不一致。
- 结算和奖励画面有愉悦感，但和“棋局结果”的关联还可以更强。
- 棋盘周围的卡片、装饰和玻璃效果有时会抢走棋盘注意力。

## 目标方向

工作方向名：**Soft Strategy Gomoku，柔和策略五子棋**

新的方向应该让用户感到：

- 亲和，但不幼稚。
- 轻松，但不松散。
- 有触感，但不是重 3D。
- 有策略气质，但不是传统老派棋类。
- 以手机体验优先，而不是桌面预览优先。
- 结算有奖励感，但不过度喧闹。

### 保留

- 浅蓝色 App 氛围。
- 米色或木质棋盘。
- 圆润卡片和舒适留白。
- 金色只用于奖励、段位提升、胜利庆祝。
- 中文 UI 文案由代码渲染，不写死在图片里。

### 强化

- 棋盘作为视觉中心。
- 当前回合、威胁、胜负状态的识别效率。
- 落子反馈。
- 结算画面和棋局结果之间的联系。
- 触控目标尺寸和安全区适配。

### 减少

- 对局页里的漂浮装饰。
- 移动端过大的页面标题。
- emoji 或依赖系统字体的图标。
- 过强的玻璃卡片，让棋盘看起来像嵌入式预览。

## 分页面优化计划

## 1. 模式选择页

### 目标

保留当前轻松亲和的第一印象，但让移动端更紧凑、更快进入游戏。

### 优化项

- 移动端压缩顶部标题区域。
- 让主要操作按钮更早出现在首屏。
- 将 emoji 风格的模式图标替换为 SVG 或 Lucide 风格图标。
- 模式卡片保持足够大的触控面积，但减少过度垂直内边距。
- 预览页里的风格标签不要进入正式生产 UI。

### 验收标准

- 390px 宽度下，不需要过度滚动即可看到前两个模式卡和设置面板标题。
- 所有模式卡片触控高度不低于 44px。
- 不再使用结构性 emoji 图标。
- 选中状态不仅靠颜色区分，也有边框、圆点或标记辅助。

## 2. 对局棋盘页

### 目标

让棋盘成为整个画面的绝对主角。

### 优化项

- 在视口允许时，将棋盘尺寸放大 15-25%。
- 减弱棋盘下方控制面板的视觉重量。
- 将装饰性元素从棋盘附近移开。
- HUD 保持轻量，只保留玩家、倒计时、当前回合等核心信息。
- 增加最近一步标记、胜利连线、威胁提示等状态反馈。
- 棋盘线条在手机上要有足够对比度。

### 验收标准

- 棋盘是页面中面积最大、视觉最强的元素。
- 用户一眼能识别当前玩家、倒计时和最后落子。
- 棋盘交叉点触控不困难。
- 游戏控制按钮不抢棋盘注意力。
- 开启减少动态效果后，仍能识别最后落子和胜利状态。

## 3. 联机大厅

### 目标

让联机流程简单、可靠、有安全感。

### 优化项

- 保留当前大卡片结构，但让“创建房间”和“加入房间”区别更明确。
- 增加适合复制/分享的房间码样式。
- 补齐等待对手、对手已加入、连接中等状态。
- 网络错误状态要提供明确恢复路径。

### 验收标准

- 用户能立刻判断主要操作是创建房间还是加入房间。
- 房间码清晰、可复制、易分享。
- 等待状态不像空白或卡住。
- 错误状态包含重试操作。

## 4. 段位页

### 目标

让段位系统更有成就感，但不要变成嘈杂的奖励界面。

### 优化项

- 保留奖章概念，但把 emoji 奖章替换为 SVG 或组件化图形。
- 更清楚展示当前段位和下一个目标。
- 金色只用于段位、奖励和高光，不用于普通装饰。
- 移动端统计卡片更紧凑。

### 验收标准

- 当前段位和距离下一段位的进度一眼可见。
- 金色有明确语义，不泛滥。
- 段位卡片在浅色和可能的深色模式下都保持可读。

## 5. 结算页

### 目标

让胜利、失败、平局、段位提升都更像“棋局结果”，而不是通用庆祝弹窗。

### 优化项

- 在结算背景中加入小棋盘片段、胜利连线或棋子轨迹。
- 庆祝动画保持短、清楚、可中断。
- 展示和棋局有关的数据：总步数、用时、执子方、胜利形态、评分。
- “再来一局”作为主操作。
- “返回选择”作为次操作。

### 验收标准

- 1 秒内能看懂胜负结果。
- 下一步主操作非常明确。
- 庆祝动效不遮挡关键文字。
- 结果文案不写死进图片。

## 组件规则

### 图标

- 结构性图标使用 SVG 或图标库组件。
- 模式、段位、奖杯、设置、联机等图标不使用 emoji。
- 同一级图标保持统一线宽和视觉风格。

### 卡片

- 卡片可以继续圆润柔和，但避免卡片套卡片过多。
- 对局页卡片应该比菜单页更轻。
- 棋盘不能看起来像一个嵌入式展示框，它应该是主场景。

### 按钮

- 最小高度 44px。
- 主操作根据页面使用蓝色或策略绿色。
- 危险操作使用珊瑚红或红色，并和主操作保持视觉距离。
- 点击反馈使用 `scale(0.96-0.98)` 或透明度变化。

### 动效

动效只用于表达状态，不做纯装饰：

- 落子动画。
- 最近一步光环。
- 胜利连线绘制。
- 弹窗进入。
- 段位提升庆祝。

动效时间建议：

- 微交互：120-240ms。
- 结算弹窗：240-420ms。
- 胜利连线：400-600ms。

必须支持 `prefers-reduced-motion`。

## 实施阶段

### Phase 1：预览页精修

范围：

- 优化 `public/settlement/soft-party/app-preview.html`。
- 移除 emoji 风格结构图标。
- 增加一版更强的对局页视觉方案。
- 收紧移动端首屏密度。

交付物：

- 更新后的独立预览页。
- 桌面截图。
- 390px 手机截图。

### Phase 2：Token 抽取

范围：

- 把确认后的视觉决策沉淀为 token。
- 在 `src/styles/tokens.css` 增加策略棋盘层 token。
- 避免组件里散落大量硬编码颜色。

交付物：

- Token 更新。
- 简短 token 使用说明。

### Phase 3：低风险页面迁移

范围：

- 先迁移模式选择页。
- 再迁移联机大厅外壳。
- 不改游戏逻辑。

交付物：

- 更新后的模式选择页。
- 更新后的联机大厅页。
- 视觉 QA 截图。

### Phase 4：对局页迁移

范围：

- 更新棋盘容器和游戏 HUD。
- 保留棋盘坐标和落子逻辑。
- 增强最后落子、胜利连线和回合状态反馈。

交付物：

- 更新后的对局页。
- 键盘与触控可访问性检查。
- 手机视口截图。

### Phase 5：结算系统集成

范围：

- 集成胜利、失败、平局、段位提升视觉系统。
- 文案继续由 React 渲染，不写死在图片素材里。
- 验证所有结算结果。

交付物：

- 更新后的结算组件。
- 各结果状态截图。
- 减少动态效果模式验证。

## QA 验收清单

- 390px 手机视口无横向滚动。
- 844px 手机高度下，主操作可以舒适触达。
- 所有触控目标不小于 44px。
- 对局页棋盘始终是视觉中心。
- 主要文字对比度达到 4.5:1。
- 仅图标按钮有可访问标签。
- 动效支持减少动态效果。
- 不使用 emoji 作为结构性图标。
- 中文 UI 文案不写进图片。
- 除非明确规划，否则不改现有游戏逻辑和路由。

## 最终建议

不建议推翻当前 `soft-party` 方向。它已经有一个很好的亲和基础。

更合适的做法是：**把 soft-party 作为菜单、联机、段位、结算的情绪层；把真正对局页升级成 soft-strategy 层。**

也就是：

- 非对局页面：保持温柔、轻松、软萌。
- 对局页面：更聚焦棋盘、更清楚、更有策略桌游感。
- 结算页面：保留奖励感，但让结果和棋局本身更强绑定。

一句话总结：  
**保留可爱，但让它更像棋。**
